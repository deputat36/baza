# Настройка закрытой Google Таблицы

## Зачем нужна закрытая таблица

Репозиторий `deputat36/baza` публичный. В нём можно хранить структуру, шаблоны и обезличенные черновики.

Реальные рабочие данные нужно хранить в закрытой Google Таблице:

- телефоны сотрудников;
- внутренние контакты центрального офиса;
- контакты партнёров;
- реальные ФИО ответственных;
- условия сотрудничества;
- комментарии по качеству подрядчиков;
- актуальные банковские контакты;
- закрытые данные по новостройкам.

## Рекомендуемое название

`База знаний специалиста | Этажи Борисоглебск`

## Рекомендуемые листы

1. Главная.
2. Контакты.
3. Ситуации.
4. База знаний.
5. Практика.
6. Предложения изменений.
7. Справочники.
8. Внутренние роли.
9. Новостройки.
10. Подрядчики.

## Права доступа

Рекомендуемая схема:

- менеджер отдела — владелец или редактор;
- ответственный за базу — редактор;
- юрист — редактор своих разделов;
- ипотечный брокер — редактор ипотечного раздела;
- сотрудники — просмотр или комментирование;
- новички — просмотр.

Редактирование всеми сотрудниками лучше не включать сразу. Иначе база быстро потеряет структуру.

## Как вносить реальные контакты

1. Импортировать безопасные CSV из GitHub.
2. В закрытой таблице заполнить реальные телефоны и ФИО.
3. Поставить статус `CHECK` для непроверенных записей.
4. После звонка или проверки заменить статус на `VERIFIED`.
5. Указать дату проверки.
6. Указать, кто проверил.

## Что нельзя вносить даже в закрытую таблицу

Не нужно хранить:

- паспортные данные клиентов;
- сканы документов;
- номера договоров по конкретным сделкам;
- банковские решения клиентов;
- переписки клиентов;
- пароли;
- токены;
- данные доступа к CRM.

Эта база предназначена для знаний и контактов, а не для хранения сделок.

## Защита листов

Рекомендуется защитить:

- справочники;
- формулы;
- ID;
- системные поля;
- лист `Главная`;
- утверждённые инструкции.

## Цветовые статусы

- `VERIFIED` — зелёный.
- `CHECK` — жёлтый.
- `DRAFT` — серый.
- `OUTDATED` — красный.
- `PRIVATE` — фиолетовый.

## Регламент актуализации

Раз в неделю:

- проверять новые предложения;
- исправлять ошибки;
- переносить полезную практику;
- отмечать устаревшие записи.

Раз в месяц:

- проверять топ-30 контактов;
- обновлять банки;
- обновлять новостройки;
- проверять подрядчиков;
- удалять дубли.

## Связь GitHub и Google Таблицы

GitHub хранит структуру и безопасные шаблоны. Google Таблица хранит рабочую версию.

Из Google Таблицы не нужно автоматически публиковать данные обратно в GitHub, пока в ней есть реальные контакты и внутренние комментарии.
