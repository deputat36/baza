# Ближайшие действия

## Главная задача

Перейти от структуры в GitHub к первой рабочей закрытой Google Таблице и недельному офисному пилоту.

## Шаг 1. Собрать рабочую таблицу

Нужно создать Google Таблицу с листами:

- Главная;
- Контакты;
- Ситуации;
- База знаний;
- FAQ;
- Практика;
- Предложения изменений;
- Справочники;
- Новостройки;
- Подрядчики;
- Обучение.

## Шаг 2. Импортировать данные

Использовать файлы из:

- `templates`;
- `data/dictionaries`;
- `data/drafts`.

Начинать лучше с минимального набора для теста.

## Шаг 3. Заполнить реальные рабочие данные

В закрытой таблице нужно заполнить:

- телефоны;
- адреса;
- графики работы;
- ответственных;
- проверенные комментарии;
- актуальные условия;
- даты проверки.

Эти данные не должны попадать в публичный GitHub.

## Шаг 4. Провести первый тест

Подключить 2–3 сотрудников.

Использовать:

- `docs/first-user-test.md`;
- `docs/office-pilot-plan.md`;
- `docs/office-demo-script.md`;
- `data/drafts/feedback-log-template.csv`.

Перед тестом запустить:

```bash
make pipeline-health
make preflight
```

## Шаг 5. Упростить структуру

После теста убрать лишнее:

- поля, которые никто не использует;
- дубли;
- непонятные формулировки;
- слишком длинные инструкции.

Отдельно добавить синонимы, которыми сотрудники реально пользуются при поиске.

## Шаг 6. Провести недельный офисный пилот

После исправлений показать таблицу на планёрке и дать 3–5 сотрудникам неделю тестового использования.

Пилот проводится по документу:

- `docs/office-pilot-plan.md`.

Цель пилота — понять, находит ли сотрудник рабочий ответ за 1–3 минуты и можно ли расширять базу на весь офис.

## Шаг 7. Принять решение о развитии

Через 2–4 недели оценить:

- оставить Google Таблицу;
- сделать закрытый портал;
- перенести в CRM;
- связать с обучением новичков;
- добавить поиск и аналитику.

Решение должно опираться на факты: найденные ответы, время поиска, количество полезных правок, повторяющиеся вопросы менеджеру и отсутствие нарушений приватности.
