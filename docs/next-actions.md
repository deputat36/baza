# Ближайшие действия

## Главная задача

Проверить новую архитектуру обучения в сборке, подготовить закрытые мастер-материалы и провести двухнедельный пилот объединенной системы.

## Шаг 1. Проверить техническую сборку

После добавления этапов, компетенций, маршрутов и модулей выполнить:

```bash
make pipeline-health
make preflight
```

Проверить:

- CSV проходят структурную проверку;
- новые источники попали в импортный план;
- лист `Обучение` появился в XLSX;
- обучающие таблицы отображаются в HTML-превью;
- в публичных артефактах нет приватных данных;
- pipeline-health не показывает критических рассинхронизаций.

## Шаг 2. Проверить обучение глазами пользователя

Открыть:

- `docs/training-system-architecture.md`;
- `data/drafts/training-learning-paths.csv`;
- `data/drafts/training-modules.csv`;
- `data/drafts/training-assessments.csv`;
- `data/drafts/training-materials-registry.csv`.

Ответить на вопросы:

1. Понимает ли новичок, с чего начать?
2. Видит ли СПН 2-3 месяцев свой маршрут стабилизации?
3. Не повторяет ли опытный СПН новичковый курс?
4. Понимает ли МОП, что проверять?
5. Есть ли у каждого модуля практика и следующий шаг?
6. Не дублирует ли учебная система факты из `Космоса`?

## Шаг 3. Подготовить закрытый Drive

Создать закрытую структуру мастер-материалов:

```text
00 Управление системой
01 Кандидат и EVP
02 Первый день
03 Быстрый старт 30 дней
04 СПН 2-3 месяца
05 СПН 4-6 месяцев
06 СПН 6+
07 Наставники и МОП
08 Юридический блок
09 Ипотека
10 Новостройки и специализации
11 УТП и продажи
99 Архив
```

Для каждого материала определить:

- единственную мастер-версию;
- владельца;
- дату проверки;
- право использования;
- аудиторию;
- статус `DRAFT`, `CHECK`, `VERIFIED` или `OUTDATED`.

Исходные внутренние материалы и файлы с ограничением копирования не загружать в публичный GitHub.

## Шаг 4. Утвердить локальный словарь Космоса

Нужно подтвердить актуальные действия и названия в CRM:

- заявка продавца;
- заявка покупателя;
- объект;
- подборка;
- встреча;
- показ;
- следующий шаг;
- задача;
- юридическая передача;
- ипотечная передача;
- задаток;
- сделка;
- постобслуживание.

После этого заменить исторические упоминания РИЕС и Echat в действующих учебных материалах.

Не требуется переписывать архивные источники. Они должны быть помечены как исторические и использоваться только для разработки новых модулей.

## Шаг 5. Утвердить локальное EVP

Проверить отдельно:

- федеральные преимущества компании;
- реальные преимущества офиса Борисоглебска;
- условия для новичка;
- условия для опытного специалиста;
- поддержку, обучение и маркетинг;
- возможности развития;
- ограничения и личную ответственность сотрудника.

Для каждого обещания зафиксировать доказательство, дату проверки и разрешенную формулировку.

## Шаг 6. Утвердить локальный каталог УТП

Для каждого доступного в Борисоглебске УТП подготовить:

- потребность клиента;
- короткую формулировку для звонка;
- полную формулировку для встречи;
- доказательство;
- ограничение применимости;
- запрещенную формулировку;
- владельца и дату проверки.

Цифры, скидки, гарантии, сроки, рейтинги и количество городов не использовать без актуального подтверждения.

## Шаг 7. Проверить обязательный маршрут первого месяца

В первую очередь утвердить модули:

1. Профессия и реалистичные ожидания.
2. Этика и безопасность.
3. Первый вход в `Космос`.
4. Поиск ответа в базе знаний.
5. Контакт с продавцом.
6. Контакт с покупателем.
7. Ценность риелтора и УТП.
8. Работа с возражениями.
9. Встреча с продавцом.
10. Объект и медиаконтент.
11. Подборка и обратная связь.
12. Показ.
13. Юридическая маршрутизация.
14. Ипотечная маршрутизация.
15. Первая воронка.

Каждый модуль должен завершаться практикой, проверкой качества и одним следующим шагом.

## Шаг 8. Провести двухнедельный пилот

Использовать:

- `docs/training-pilot-plan.md`.

Минимальная группа:

- один новичок;
- один СПН 2-6 месяцев;
- один опытный СПН;
- один МОП;
- профильный эксперт по необходимости.

Проверить:

- понятность маршрута;
- время поиска материала;
- качество практических заданий;
- соответствие `Космосу`;
- отсутствие двойного ввода;
- число устаревших формулировок;
- фактическую помощь сотруднику.

## Шаг 9. Упростить систему после пилота

Удалить или объединить:

- модули без наблюдаемого результата;
- дублирующие материалы;
- устаревшие инструкции;
- ручные отчеты, повторяющие `Космос`;
- обязательные поля, которые не влияют на решение;
- длинные презентации, которые можно заменить короткой практикой.

## Шаг 10. Принять решение о масштабировании

Систему можно расширять на офис, если:

- сотрудник открывает один маршрут по своей роли;
- обязательные модули первого месяца проверены;
- операционные факты берутся из `Космоса`;
- локальные EVP и УТП утверждены;
- мастер-материалы находятся в закрытом Drive;
- менеджер видит только блокер, помощь и следующий шаг;
- нарушений приватности нет.
