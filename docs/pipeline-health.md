# Pipeline health

## Назначение

`pipeline-health` — это техническая проверка согласованности проекта.

Она нужна, чтобы после добавления новых генераторов, отчетов, CSV-источников и workflow не расходились между собой:

- `Makefile`;
- `.github/workflows/validate-data.yml`;
- скрипты в `scripts/tools/`;
- `README.md`;
- preflight-сводка;
- индекс артефактов сборки.

Проект уже содержит много генераторов. Без отдельной проверки легко получить ситуацию, когда новый отчет есть в одном месте, но не попадает в GitHub Actions, не запускается через `make preflight` или не отображается в артефактах.

## Как запустить

Отдельно:

```bash
make pipeline-health
```

В составе полной проверки:

```bash
make preflight
```

В GitHub Actions отчет создается автоматически в workflow `Validate data files`.

## Что создается

После запуска появляются файлы:

- `build/pipeline-health-report.md` — человекочитаемый отчет;
- `build/pipeline-health-report.csv` — табличная версия для анализа и фильтрации.

## Что проверяется

Отчет проверяет:

1. Наличие обязательных управляющих файлов проекта.
2. Наличие ключевых целей в `Makefile`.
3. Существование скриптов, на которые ссылаются `Makefile` и workflow.
4. Расхождения между скриптами в `Makefile` и `.github/workflows/validate-data.yml`.
5. Наличие важных маркеров в `README.md`.
6. Наличие ключевых артефактов в `build_preflight_summary.py`.
7. Наличие ключевых артефактов в `build_artifact_index.py`.
8. Наличие итоговых целей `summary`, `artifact-index`, `pipeline-health` и `inventory` в цепочке `preflight`.

## Статусы

| Статус | Значение | Что делать |
|---|---|---|
| `OK` | Проверка прошла | Ничего не требуется |
| `WARN` | Есть потенциальная рассинхронизация | Проверить вручную и решить, нужно ли править |
| `FAIL` | Есть отсутствующий файл, цель или скрипт | Исправить до передачи артефактов сотрудникам |

## Когда обязательно запускать

Запускайте `make pipeline-health` после изменений в любом из блоков:

- добавлен новый генератор отчета;
- изменен `Makefile`;
- изменен workflow GitHub Actions;
- изменена preflight-сводка;
- изменен индекс артефактов;
- добавлен новый обязательный документ в `docs/`;
- меняется структура build-артефактов.

## Практический смысл для офиса

Для сотрудников база должна выглядеть как надежный инструмент, а не как набор файлов.

Если менеджер открывает артефакты сборки, он должен видеть понятный и актуальный набор отчетов. Если разработчик добавляет новый генератор, он должен попадать в общий маршрут проверки. `pipeline-health` помогает держать этот маршрут под контролем.

## Ограничения

Проверка не заменяет полный запуск `make preflight` и не оценивает качество содержания базы знаний.

Она отвечает только за техническую согласованность сборочного контура.
