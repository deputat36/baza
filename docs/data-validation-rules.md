# Правила проверки данных

## Цель

Перед переносом записи в рабочую таблицу нужно проверить, что она понятна, безопасна и пригодна для использования.

## Минимальный набор полей

Для любой записи обязательны:

- ID;
- тип;
- категория;
- название или организация;
- краткое назначение;
- приоритет;
- статус.

## Проверка безопасности

Запись не должна содержать:

- данные клиентов;
- паспортные данные;
- закрытые условия сделки;
- пароли и ключи;
- внутренние телефоны в публичной версии.

## Проверка пользы

Запись должна отвечать хотя бы на один вопрос:

- куда обращаться;
- что делать;
- какой документ нужен;
- какая частая ошибка;
- кого подключить;
- какой риск проверить.

## Проверка понятности

Запись должна быть понятна новому сотруднику.

Если для понимания нужен отдельный комментарий менеджера, запись нужно переписать проще.

## Проверка статуса

- `DRAFT` — запись можно хранить как черновик.
- `CHECK` — нужно проверить перед использованием.
- `VERIFIED` — можно использовать в работе.
- `OUTDATED` — не использовать.
- `PRIVATE` — хранить только в закрытой таблице.

## Правило приоритета

Приоритет 5 ставится только тем записям, которые часто влияют на сделку или безопасность процесса.

Приоритет 1 ставится справочным материалам, которые редко используются.
