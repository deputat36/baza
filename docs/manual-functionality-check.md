# Ручная проверка функционала проекта

## Назначение

Этот документ нужен, чтобы быстро проверить, что проект можно открыть и использовать как рабочий прототип базы знаний.

Проверка не требует реальных контактов, клиентских данных и закрытых условий.

## Что сейчас можно проверить

На текущем этапе можно проверить:

- сборку HTML-интерфейса;
- стартовую страницу артефактов;
- HTML-превью CSV-структуры;
- поиск по HTML-превью;
- сборку XLSX;
- управленческие отчёты;
- пакет запуска;
- будущие контракты совместимости;
- сценарии подготовки закрытой Google Таблицы.

## Вариант 1. Проверка через GitHub Pages

1. Открыть репозиторий `deputat36/baza`.
2. Перейти в `Settings -> Pages`.
3. Установить `Source: GitHub Actions`.
4. Перейти в `Actions`.
5. Запустить workflow `Publish HTML preview`.
6. Дождаться успешного завершения.
7. Открыть ссылку Pages.

Ожидаемый адрес обычно выглядит так:

```text
https://deputat36.github.io/baza/
```

Точный адрес нужно смотреть в результате workflow или в настройках GitHub Pages.

## Вариант 2. Проверка через артефакт GitHub Actions

1. Перейти в `Actions`.
2. Запустить workflow `Validate data files`.
3. Дождаться успешного завершения.
4. Скачать артефакт `baza-build-artifacts`.
5. Распаковать архив.
6. Открыть файл:

```text
index.html
```

## Что открыть первым

После сборки первым открыть:

```text
index.html
```

Дальше проверить:

```text
html-preview/index.html
manager-dashboard.md
office-launch-packet.md
go-no-go-report.md
launch-decision-log.md
weekly-manager-digest.md
manager-action-register.md
```

## Что проверить в HTML-интерфейсе

1. Открывается ли стартовая страница `index.html`.
2. Есть ли ссылки на основные отчёты.
3. Открывается ли `html-preview/index.html`.
4. Работает ли поиск по CSV-разделам.
5. Есть ли быстрые ссылки по разделам.
6. Отображаются ли карточки CSV.
7. Видны ли предупреждения о приватных данных.
8. Не попали ли в публичный просмотр реальные телефоны, ФИО и клиентские данные.

## Что проверить в управленческих отчётах

1. `manager-dashboard.md` — есть ли краткая картина состояния проекта.
2. `office-launch-packet.md` — есть ли порядок запуска.
3. `go-no-go-report.md` — есть ли решение `GO` или `NO-GO`.
4. `manager-action-register.md` — есть ли список действий менеджера.
5. `role-navigation-report.md` — понятны ли маршруты по ролям.
6. `section-ownership-report.md` — понятны ли зоны ответственности.
7. `office-acceptance-test-report.md` — есть ли проверочные сценарии.

## Что пока не является готовым функционалом

Пока не готово как полноценный рабочий инструмент:

- закрытая Google Таблица с реальными контактами;
- заполненные реальные кураторы и городские контакты;
- боевое подключение к Навигатору сделок;
- автоматическое определение подсказок по реальным сделкам;
- авторизация сотрудников;
- серверный личный кабинет.

## Критерий успешной проверки

Проверка считается успешной, если:

- GitHub Actions завершился без ошибок;
- `index.html` открывается;
- `html-preview/index.html` открывается;
- поиск в HTML-превью работает;
- основные отчёты открываются;
- в публичных артефактах нет приватных данных;
- понятно, какие следующие шаги нужны для закрытой Google Таблицы.

## Что делать при ошибке

Если workflow упал:

1. Открыть ошибочный шаг.
2. Скопировать traceback или текст ошибки.
3. Передать ошибку в этот диалог.
4. Не вносить реальные контакты в публичный репозиторий для теста.
