# Как сотрудник будет пользоваться базой

## Главное

Сотрудник не работает с GitHub.

GitHub нужен руководителю и ответственному за проект как хранилище структуры, правил и шаблонов.

Сотрудник работает с приватной Google Таблицей, собранной на основе файлов из репозитория.

## Рабочий сценарий

### 1. У сотрудника появляется вопрос

Например:

- клиент продаёт квартиру с детскими долями;
- покупатель идёт в ипотеку;
- продавец потерял документы;
- нужно понять, куда обращаться по перепланировке;
- клиент спрашивает про долги ЖКХ;
- нужно передать квартиру после сделки.

### 2. Сотрудник открывает Google Таблицу

На главной странице он видит:

- быстрые ссылки на разделы;
- частые ситуации;
- поиск;
- последние изменения;
- контакты ответственных.

### 3. Сотрудник ищет обычными словами

Не нужно знать точное название документа или отдела.

Можно искать так:

- `детские доли`;
- `опека`;
- `ЕГРН`;
- `доверенность`;
- `ипотека`;
- `маткапитал`;
- `передача квартиры`;
- `долги ЖКХ`.

Для этого в базе есть отдельный словарь ключевых слов.

### 4. Сотрудник открывает нужную запись

В записи он видит:

- что это за ситуация;
- когда она применяется;
- что делать по шагам;
- какие документы нужны;
- кого подключить;
- куда обращаться;
- частые ошибки;
- важные предупреждения;
- связанные контакты и документы;
- статус актуальности.

### 5. Сотрудник принимает рабочее решение

Например:

- подключить юриста до аванса;
- запросить выписку ЕГРН;
- отправить клиента к ипотечному брокеру;
- уточнить документы по опеке;
- проверить задолженность;
- подготовить акт приёма-передачи.

## Пример 1. Квартира с детскими долями

Сотрудник ищет `детские доли`.

Открывает ситуацию `Продажа квартиры с детскими долями`.

Видит:

- нужен юрист;
- нужна опека;
- нужны документы по детям;
- нельзя обещать сроки без проверки;
- связанный документ — разрешение опеки;
- связанный контакт — органы опеки.

Результат: сотрудник не берёт аванс вслепую и не обещает клиенту лишнего.

## Пример 2. Покупатель с ипотекой

Сотрудник ищет `ипотека`.

Открывает банковский или ипотечный блок.

Видит:

- кого подключить;
- какие банки проверить;
- нужна ли оценка;
- какие документы по объекту понадобятся;
- какие частые ошибки бывают.

Результат: ипотечный брокер подключается вовремя, а сделка не тормозится из-за неподготовленного объекта.

## Пример 3. Утерян документ

Сотрудник ищет `потеряли документы` или `утерян документ`.

Открывает ситуацию по восстановлению документов.

Видит:

- какие документы можно запросить;
- где искать информацию;
- когда нужен Росреестр;
- когда нужен архив;
- когда нужен МФЦ;
- какие данные запросить у клиента.

Результат: сотрудник понимает, с чего начать, и не гоняет клиента по кругу.

## Пример 4. Передача квартиры

Сотрудник ищет `передача квартиры`.

Открывает ситуацию и шаблон акта.

Видит:

- проверить ключи;
- снять показания счётчиков;
- проверить квитанции;
- зафиксировать состояние квартиры;
- подписать акт.

Результат: после сделки меньше бытовых конфликтов между продавцом и покупателем.

## Что видит новичок

Новичок использует базу как учебник-подсказку.

Он не обязан помнить все нюансы, но должен научиться:

- быстро находить ситуацию;
- понимать порядок действий;
- видеть, кого подключить;
- не пропускать частые ошибки;
- задавать правильные вопросы клиенту.

## Что видит опытный специалист

Опытный специалист использует базу как справочник и место для накопления практики.

Он может:

- быстро проверить контакты;
- добавить практический совет;
- предложить исправление;
- оценить партнёра;
- помочь улучшить инструкцию.

## Что делает менеджер

Менеджер видит:

- какие разделы заполнены;
- какие данные требуют проверки;
- кто вносит полезные правки;
- какие ситуации чаще всего вызывают вопросы;
- где новичкам не хватает информации.

## Что делает ответственный за базу

Ответственный:

- проверяет предложения сотрудников;
- обновляет статусы;
- убирает дубли;
- переносит полезную практику;
- следит, чтобы в базе не было лишних и опасных данных.

## Итоговая логика

База должна работать как навигатор по сделке.

Сотрудник не читает её подряд.

Он открывает базу в момент вопроса, ищет свою ситуацию, получает короткий порядок действий и понимает следующий шаг.
