# Контроль актуальности данных

## Назначение

Контроль актуальности нужен, чтобы база знаний не превращалась в набор устаревших контактов и инструкций.

Он фиксирует:

- какие разделы нужно регулярно проверять;
- с какой периодичностью;
- кто отвечает за проверку;
- когда данные проверялись в последний раз;
- какие разделы требуют внимания.

## Файлы

- `data/dictionaries/freshness-policies.csv` — политики актуальности.
- `data/drafts/data-freshness-register.csv` — реестр контролируемых разделов.

## Проверка

```bash
make freshness
```

или напрямую:

```bash
python scripts/tools/build_data_freshness_report.py
```

## Результаты

Скрипт создаёт:

- `build/data-freshness-report.md`;
- `build/data-freshness-report.csv`.

## Состояния

- `OK` — проверка ещё актуальна.
- `DUE_SOON` — скоро нужна повторная проверка.
- `OVERDUE` — срок проверки прошёл.
- `NEVER_CHECKED` — дата последней проверки не указана.
- `INVALID_DATE` — дата заполнена в неправильном формате.

## Важно

Просроченная проверка не ломает GitHub Actions. Это рабочий сигнал для менеджера, а не техническая ошибка.

CI падает только при структурных ошибках: нет файла, нет политики, неверная дата, дубль ID или отсутствует обязательное поле.
