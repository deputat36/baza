# Управленческая сводка базы знаний

## Назначение

Dashboard нужен менеджеру, чтобы быстро увидеть состояние базы знаний без ручного чтения всех технических отчётов.

Он собирает в одном месте:

- готовность разделов;
- актуальность данных;
- ответственность за разделы;
- открытые предложения и исправления;
- пустые обязательные поля;
- риски будущей интеграции.

## Команда

```bash
make manager-dashboard
```

Обычно отдельно запускать не нужно: dashboard входит в `make preflight`.

## Результаты

Скрипт создаёт:

- `build/manager-dashboard.md`;
- `build/manager-dashboard.csv`.

## Как читать

- `OK` — явного управленческого риска по метрике нет.
- `WARN` — нужно ручное внимание менеджера или ответственного.

## Что контролировать

1. Разделы со статусами `CHECK` и `DRAFT`.
2. Контакты и инструкции без даты последней проверки.
3. Критичные зоны ответственности без статуса `READY`.
4. Открытые заявки сотрудников с приоритетом 4-5.
5. Пустые ID, статусы и приоритеты.
6. Интеграционные доступы к закрытым данным.
7. Контракты будущей интеграции, где не найден источник или артефакт.

## Управленческий вывод

Если dashboard показывает много `WARN`, значит перед запуском базы нужно не добавлять новые функции, а закрыть базовые проблемы данных и ответственности.

Если `WARN` мало, следующий шаг — ручная проверка смысла записей и перенос рабочих данных в закрытую Google Таблицу.

## Ограничение

Dashboard агрегирует технические отчёты. Он не подтверждает юридическую точность, актуальность реальных контактов и допустимость публикации данных.
