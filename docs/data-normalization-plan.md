# План нормализации данных

## Зачем нужна нормализация

Перед импортом в рабочую Google Таблицу нужно привести данные к единому виду. Иначе сотрудники будут искать одно и то же разными способами, а база быстро превратится в набор разрозненных списков.

## Что проверить в первую очередь

### ID

У каждой записи должен быть уникальный ID.

Примеры:

- `ORG-0301` — организация;
- `DOC-0001` — документ;
- `SIT-0001` — ситуация;
- `BNK-0401` — банк;
- `FAQ-0001` — вопрос;
- `TMP-0010` — шаблон;
- `TRN-0001` — обучающий чек-лист.

### Категории

Категории должны совпадать со справочником `data/dictionaries/categories-core.csv`.

Не нужно создавать новые категории без необходимости. Лучше добавить подкатегорию.

### Статусы

Использовать только статусы из `data/dictionaries/statuses.csv`.

Основные статусы:

- `DRAFT`;
- `CHECK`;
- `VERIFIED`;
- `OUTDATED`;
- `PRIVATE`.

### Связанные записи

В поле связанных записей использовать ID, а не произвольный текст.

Правильно:

`DOC-0001; ORG-0302; SIT-0003`

Неправильно:

`выписка, мфц, проверка объекта`

## Единый стиль описаний

Фразы должны быть короткими и рабочими.

Лучше:

`Перед авансом проверить выписку ЕГРН и состав собственников.`

Хуже:

`В принципе желательно было бы посмотреть документы, если получится.`

## Какие поля можно скрыть в рабочей таблице

Для сотрудников можно скрыть:

- источник;
- дата проверки;
- проверил;
- технический тип;
- внутренние связи;
- статус безопасности.

Для ответственного за базу эти поля должны оставаться доступными.

## Что объединить

При чистке данных проверить дубли:

- МФЦ как контакт и как организация;
- Росреестр как контакт и как сервис;
- нотариус как раздел и как конкретная контора;
- ипотека как ситуация и как банковский блок;
- передача квартиры как ситуация и как шаблон акта.

Дубли не всегда нужно удалять. Иногда лучше связать записи между собой.

## Минимальная проверка перед публикацией

Перед переводом записи в `VERIFIED` проверить:

1. Есть понятное название.
2. Есть категория.
3. Есть описание, когда использовать.
4. Есть статус.
5. Есть дата проверки.
6. Нет персональных данных клиентов.
7. Нет закрытых условий.
8. Есть связанные записи, если они нужны.

## Результат

После нормализации база должна быть удобна для поиска, фильтров и будущего переноса в CRM или отдельный портал.
