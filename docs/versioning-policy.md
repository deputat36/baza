# Политика версий базы знаний

## Зачем нужна версия

Версия нужна, чтобы понимать, какая структура сейчас считается актуальной и что изменилось после тестов.

Проект развивается в два слоя:

1. Публичный репозиторий — структура, шаблоны, безопасные черновики.
2. Закрытая Google Таблица — реальные контакты, проверенные данные и рабочие комментарии.

## Рекомендуемая схема версий

Использовать формат:

`v0.1`, `v0.2`, `v0.3`, `v1.0`

## Значения версий

### v0.1 — архитектурный MVP

Есть структура, документы, шаблоны и черновики.

Нет полноценной рабочей таблицы с проверенными контактами.

### v0.2 — табличный MVP

Создана закрытая Google Таблица.

Импортированы основные разделы.

Заполнены первые реальные контакты в закрытом доступе.

### v0.3 — тестовый MVP

Таблицу протестировали 2–3 сотрудника.

Есть обратная связь.

Исправлены первые ошибки структуры.

### v1.0 — рабочая версия офиса

База используется сотрудниками регулярно.

Есть ответственные за разделы.

Работает еженедельное обслуживание.

Доля проверенных записей растёт.

## Что фиксировать при изменениях

В `changelog.md` фиксировать:

- добавленные файлы;
- изменённые разделы;
- новые шаблоны;
- крупные структурные решения;
- запуск тестов;
- результаты тестов;
- переход к новой версии.

## Что не фиксировать в публичном changelog

Не нужно указывать:

- реальные телефоны;
- ФИО клиентов;
- подробности сделок;
- закрытые условия;
- приватные комментарии по людям.

## Текущая рекомендуемая версия

Текущий проект соответствует уровню `v0.1`.

Следующая цель — перейти к `v0.2`, то есть собрать закрытую Google Таблицу и импортировать первый набор данных.
