# План офисного пилота базы знаний

## Назначение

Офисный пилот нужен, чтобы аккуратно перейти от архитектурного MVP к реальному использованию базы знаний сотрудниками.

Это не полноценный запуск на весь офис. Это короткая проверка на ограниченной группе, которая показывает, помогает ли база в ежедневной работе и какие разделы нужно упростить до общего запуска.

## Главный принцип

Пилот должен проверять не красоту структуры, а практическую пользу.

Сотрудник открывает базу, вводит рабочий вопрос и должен быстро понять:

- что делать дальше;
- какие документы или данные нужны;
- кого подключить;
- какие риски есть;
- где оставить замечание, если ответа нет.

## Когда можно запускать пилот

Пилот можно запускать, когда выполнены минимальные условия:

1. Создана закрытая Google Таблица.
2. Импортирован минимальный набор листов.
3. Не публикуются реальные внутренние контакты и клиентские данные в публичном репозитории.
4. Есть понятная инструкция входа для сотрудника.
5. Есть форма или лист для фиксации обратной связи.
6. Менеджер понимает, кто принимает решения по правкам.
7. Перед выдачей сотрудникам запущены базовые проверки проекта.

Рекомендуемые команды перед пилотом:

```bash
make pipeline-health
make preflight
```

## Минимальный набор данных для пилота

Для первого пилота не нужно заполнять всё.

Достаточно подготовить рабочий минимум:

| Блок | Что должно быть готово | Зачем нужно |
|---|---|---|
| Ситуации | 15–25 частых ситуаций | Проверить поиск по реальным вопросам сотрудников |
| База знаний | Краткие инструкции по ключевым темам | Дать порядок действий, а не просто справочник |
| Документы | Базовые списки документов по сделкам | Помочь новичку не забыть обязательные документы |
| FAQ | 20–30 частых вопросов | Быстро закрывать типовые обращения |
| Синонимы поиска | Слова, которыми реально пользуются сотрудники | Найти ответ не только по официальной формулировке |
| Предложения изменений | Простой способ оставить замечание | Не потерять обратную связь после теста |

Контакты, телефоны, условия банков и данные партнёров заполняются только в закрытой рабочей таблице.

## Участники пилота

Оптимальный состав — 3–5 человек.

| Роль | Задача |
|---|---|
| Менеджер | Запускает пилот, собирает обратную связь, принимает решение |
| Новичок | Проверяет, понятна ли база человеку без опыта |
| Опытный специалист | Проверяет практическую точность и полноту |
| Сотрудник, часто работающий с ипотекой или юристом | Проверяет маршрутизацию сложных вопросов |
| Ответственный за данные | Вносит безопасные правки в структуру и закрытую таблицу |

В публичных документах не нужно фиксировать ФИО участников. Достаточно ролей.

## План на неделю

### День 0. Подготовка

1. Создать или проверить закрытую Google Таблицу.
2. Импортировать минимальный набор данных.
3. Проверить, что публичный репозиторий не содержит приватных данных.
4. Подготовить короткую инструкцию для участников.
5. Выбрать 3–5 тестовых сценариев из `docs/first-user-test.md`.
6. Назначить место для обратной связи.

### День 1. Короткая демонстрация

На планёрке или отдельной встрече показать базу за 10–15 минут.

Объяснить только главное:

- где искать ответ;
- как пользоваться фильтрами;
- что делать, если ответ не найден;
- как оставить предложение;
- какие данные нельзя переносить в публичный репозиторий.

Не нужно долго рассказывать всю структуру. Если база требует долгого объяснения, её нужно упрощать.

### Дни 2–4. Использование в рабочих ситуациях

Участники применяют базу в реальной работе.

Каждый раз фиксируется:

- какой вопрос искали;
- найден ли ответ;
- сколько примерно занял поиск;
- что было непонятно;
- каких слов не хватило в поиске;
- какую правку нужно внести.

В этот период важно не защищать текущую структуру, а собирать честные неудобства.

### День 5. Разбор и правки

Менеджер собирает обратную связь и делит её на группы:

| Группа | Что сделать |
|---|---|
| Быстрые правки | Исправить формулировки, синонимы, дубли, лишние поля |
| Важные пробелы | Добавить недостающие ситуации и инструкции |
| Спорные правки | Передать на проверку юристу, ипотечному специалисту или ответственному |
| Идеи на потом | Перенести в очередь улучшений |

### День 6–7. Решение по запуску

По итогам недели принимается одно из решений:

| Решение | Когда выбирать | Что делать дальше |
|---|---|---|
| GO | Сотрудники находят ответы, критичных ошибок нет | Расширять использование на офис |
| GO с правками | База полезна, но есть понятные недочеты | Исправить быстрые замечания и запускать шире |
| Повторный пилот | Сотрудники теряются или не находят ответы | Упростить структуру и провести второй тест |
| NO-GO | Есть риски приватности или база не помогает работе | Остановить запуск, исправить основу |

## Метрики пилота

Минимально отслеживать:

| Метрика | Хороший ориентир |
|---|---|
| Среднее время поиска ответа | 1–3 минуты |
| Доля найденных ответов | Не менее 70% по тестовым сценариям |
| Количество полезных правок | Не менее 10 за неделю |
| Количество повторяющихся вопросов менеджеру | Должно снижаться |
| Понятность для новичка | Новичок может объяснить следующий шаг своими словами |
| Нарушения приватности | 0 |

## Красные флаги

Пилот нужно остановить или ограничить, если:

- в публичный репозиторий попали персональные данные, телефоны, закрытые условия или комментарии по сделкам;
- сотрудники не могут найти базовые ответы даже после короткого объяснения;
- в базе много неподтвержденных инструкций без статуса;
- непонятно, кто отвечает за исправления;
- сотрудники используют базу как CRM и пытаются хранить там клиентов или сделки;
- материалы противоречат юридической или ипотечной практике офиса.

## Чек-лист менеджера перед пилотом

- [ ] Закрытая таблица создана.
- [ ] Доступ выдан только нужным участникам.
- [ ] Минимальные листы импортированы.
- [ ] Реальные контакты и закрытые данные не опубликованы в GitHub.
- [ ] Есть инструкция для участников.
- [ ] Есть место для обратной связи.
- [ ] Выбраны тестовые сценарии.
- [ ] Запущены `make pipeline-health` и `make preflight`.
- [ ] Назначен ответственный за разбор правок.
- [ ] Назначена дата решения GO/NO-GO.

## Что делать после пилота

После пилота нужно не просто собрать отзывы, а превратить их в управляемые действия.

Порядок:

1. Перенести замечания в очередь изменений.
2. Разделить правки на срочные, важные и будущие.
3. Обновить синонимы поиска.
4. Упростить непонятные инструкции.
5. Добавить отсутствующие частые ситуации.
6. Проверить спорные темы с профильным специалистом.
7. Обновить статус проекта.
8. Принять решение о расширении на офис.
