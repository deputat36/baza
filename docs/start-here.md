# С чего начать просмотр проекта

## Коротко

Проект уже можно смотреть как структуру будущей базы знаний и единой системы обучения специалистов.

Готовый рабочий инструмент появится после проверки новых обучающих маршрутов, сборки закрытой Google Таблицы, размещения мастер-материалов в закрытом Drive и пилота на сотрудниках.

Операционная работа ведется в `Космосе`. Репозиторий не должен дублировать клиентов, объекты, сделки и фактическую активность сотрудников.

## Что открыть первым

1. `README.md` — общее описание.
2. `docs/project-status.md` — текущий этап и ближайшие блокеры.
3. `docs/full-audit-training-integration.md` — полный аудит проекта и материалов.
4. `docs/training-system-architecture.md` — целевая архитектура обучения.
5. `docs/project-index.md` — карта всех файлов проекта.
6. `data/drafts/training-learning-paths.csv` — маршруты по ролям и стажу.
7. `data/drafts/training-modules.csv` — практические учебные модули.
8. `docs/ui-preview.md` — как открыть пользовательский интерфейс.
9. `docs/google-sheets-mvp.md` — как устроена закрытая Google Таблица.
10. `docs/office-pilot-plan.md` — как провести офисный пилот.

## Главная логика обучения

```text
роль -> этап -> компетенция -> материал -> практика -> действие в Космосе -> подтверждение -> следующий этап
```

Чтение файла или формальная подпись не являются доказательством навыка.

Сотрудник должен:

- объяснить порядок своими словами;
- выполнить практическое задание;
- показать операционный результат в `Космосе`, если это применимо;
- не допустить критической ошибки;
- понимать следующий шаг.

## Основные файлы обучения

```text
data/dictionaries/training-stages.csv
data/dictionaries/training-competencies.csv
data/drafts/training-learning-paths.csv
data/drafts/training-modules.csv
data/drafts/training-assessments.csv
```

Существующий `data/drafts/role-training-plan.csv` отвечает только за обучение работе с базой знаний. Он не заменяет профессиональную программу риелтора.

## Что уже можно оценить без сборки

- Общую идею базы знаний.
- Разделение `Космоса`, `baza`, закрытого Drive и учебной таблицы.
- Ролевые маршруты от кандидата до СПН 6+ и наставничества.
- Словарь профессиональных компетенций.
- Практические модули и контрольные точки.
- Логику карточек контактов, ситуаций и документов.
- Правила безопасности и использования источников.
- План импорта в Google Таблицу.
- Apps Script для настройки закрытой таблицы.

## Где лежат черновики данных

```text
data/drafts/
data/dictionaries/
templates/
```

В этих файлах есть структура и безопасные черновики.

Реальные внутренние телефоны, ФИО, индивидуальные результаты, закрытые условия и клиентские данные в публичный репозиторий не добавляются.

## Где хранить исходные учебные материалы

Презентации, PDF, рабочие тетради, видео и внутренние регламенты должны храниться в закрытом Google Drive.

В публичном репозитории сохраняются только:

- безопасное название источника;
- тема;
- аудитория;
- статус актуальности;
- право использования;
- отметка о необходимости адаптации.

Материалы с запретом копирования нельзя публиковать целиком.

## Как посмотреть в браузере

Если работа идет локально:

```bash
make html-preview
```

Ожидаемый файл:

```text
build/html-preview/index.html
```

HTML-превью показывает CSV-структуру, включая новые обучающие справочники и модули.

Если в репозитории включен GitHub Pages с источником `GitHub Actions`, HTML-превью можно опубликовать через workflow:

```text
Actions -> Publish HTML preview -> Run workflow
```

Ожидаемый адрес обычно выглядит так:

```text
https://deputat36.github.io/baza/
```

Точный адрес нужно смотреть в настройках GitHub Pages или в результате workflow.

## Как посмотреть промежуточный XLSX

Если работа идет локально:

```bash
make install
make preflight
```

Ожидаемый файл:

```text
build/baza-knowledge-mvp.xlsx
```

Новые учебные файлы должны появиться на листах `Справочники` и `Обучение`.

Если сборка идет через GitHub Actions, HTML-превью и XLSX нужно искать в артефактах workflow:

```text
Actions -> Validate data files -> Artifacts -> baza-build-artifacts
```

## Как собрать закрытую Google Таблицу

1. Создать закрытую Google Таблицу.
2. Подключить Apps Script из папки `scripts/google-apps-script/`.
3. Обновить страницу таблицы.
4. Открыть меню `База знаний`.
5. Запустить `Полная подготовка таблицы`.
6. Импортировать CSV по `build/import-plan.md` или `docs/import-manifest.md`.
7. Запустить `База знаний -> 3. Проверить данные`.
8. Исправить замечания на листе `Проверка данных`.
9. Ограничить доступ к индивидуальным результатам обучения.

## Что пока не готово

- Закрытая рабочая Google Таблица с реальными контактами.
- Закрытый Drive с назначенными мастер-версиями материалов.
- Подтвержденный словарь актуальных действий в `Космосе`.
- Проверенное локальное EVP Борисоглебска.
- Утвержденный локальный каталог УТП.
- Проверка всех модулей первого месяца профильными владельцами.
- Пилот новой системы на сотрудниках.
- Финальный запуск на офис.

## Ближайший практический шаг

1. Запустить `make pipeline-health` и `make preflight`.
2. Открыть XLSX и HTML-превью.
3. Проверить маршруты и модули глазами новичка, опытного СПН и МОП.
4. Создать закрытый реестр мастер-материалов.
5. Подтвердить действия в `Космосе`.
6. Провести двухнедельный пилот.
