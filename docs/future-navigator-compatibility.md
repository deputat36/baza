# Готовность к будущей связке с Навигатором сделок

## Решение на текущем этапе

Проекты не совмещаются сейчас.

`baza` развивается как самостоятельная база знаний, но структура должна оставаться совместимой с будущим подключением к рабочей карточке сделки.

## Граница проектов

- `baza` — знания, инструкции, контакты, типовые ситуации, документы и правила.
- Навигатор сделок — конкретные сделки, участники, документы сделки, задачи, риски, комментарии и история.

Данные сделок, ФИО клиентов, телефоны участников и рабочие комментарии не должны попадать в публичный репозиторий `baza`.

## Совместимость через ID

Для будущей связки нужно сохранять стабильные ID:

- `SIT-*` — типовые ситуации;
- `DOC-*` — документы и шаблоны документов;
- `ORG-*` — организации и контакты;
- `INS-*` — инструкции;
- `FAQ-*` — вопросы и ответы;
- `PRC-*` — практика и частые ошибки.

Карточка сделки в будущем сможет ссылаться на эти ID, не копируя содержимое базы знаний внутрь сделки.

## Возможный пользовательский сценарий

В карточке сделки сотрудник видит блок `Подсказки базы знаний`.

Примеры правил:

- если в сделке есть дети, показать `SIT-0001` и `DOC-0007`;
- если нужна справка о зарегистрированных, показать `DOC-0006` и связанные контакты;
- если есть ипотека, показать ипотечные инструкции и контакты брокера;
- если есть перепланировка, показать ситуацию по перепланировке и список проверок;
- если статус сделки близок к задатку, показать документы и риски, блокирующие задаток.

## Рекомендуемая архитектура будущей интеграции

1. Оставить `baza` источником знаний.
2. Собирать `build/knowledge-index.json` как read-only индекс.
3. После проверки MVP перенести проверенные знания в закрытый источник.
4. Для Supabase использовать отдельные таблицы с префиксом `kb_*`, не смешивая их с `nav_*` и `leader_*`.
5. В Навигаторе читать подсказки через отдельный API/Edge Function, а не напрямую из публичного CSV.
6. В карточке сделки показывать только релевантные записи по ID и правилам.

## Что не делать

- Не копировать реальные сделки в базу знаний.
- Не хранить клиентские данные в `baza`.
- Не смешивать таблицы `kb_*`, `nav_*` и `leader_*`.
- Не делать Навигатор зависимым от публичного GitHub Pages как от боевого источника.
- Не подключать write-действия до появления RLS и тестов доступа.

## Ближайший безопасный шаг

Развивать качество ID, связей и статусов в `baza`, чтобы каждая полезная запись могла быть показана в будущем интерфейсе как независимая подсказка.
