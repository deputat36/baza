# План первого запуска MVP

## Цель запуска

Запустить базу знаний в минимальном рабочем виде, чтобы сотрудники могли быстро находить ответы по типовым ситуациям сделки.

MVP не должен быть идеальным. Он должен быть полезным каждый день.

## Что входит в MVP

Обязательные листы:

1. Главная.
2. Контакты.
3. Ситуации.
4. База знаний.
5. Практика.
6. Предложения изменений.
7. Справочники.

Обязательные разделы:

- государственные органы;
- МФЦ и Росреестр;
- базовые документы;
- типовые ситуации;
- банки и ипотека;
- нотариусы;
- кадастровые инженеры;
- оценщики;
- ЖКХ;
- управляющие компании;
- подрядчики;
- новостройки;
- шаблоны документов;
- FAQ;
- онлайн-сервисы.

## Что не входит в MVP

На первом этапе не нужно делать:

- отдельный сайт;
- полноценную CRM-интеграцию;
- сложные права доступа;
- автоматическую синхронизацию;
- личные карточки клиентов;
- хранение документов по сделкам;
- публикацию внутренних телефонов в GitHub;
- публикацию закрытых условий компании.

Сначала нужно доказать, что база действительно нужна сотрудникам.

## Этап 1. Подготовка таблицы

1. Создать закрытую Google Таблицу.
2. Импортировать CSV из папок `templates` и `data/drafts`.
3. Настроить фильтры.
4. Закрепить первую строку.
5. Скрыть технические поля, если они мешают сотрудникам.
6. Добавить цветовую маркировку статусов.
7. Добавить лист `Справочники` для выпадающих списков.
8. Настроить доступ только нужным сотрудникам.

## Этап 2. Проверка данных

Перед выдачей сотрудникам нужно проверить:

- телефоны;
- адреса;
- графики работы;
- ответственных;
- актуальность банковских условий;
- список нотариусов;
- список кадастровых инженеров;
- разделы ЖКХ и УК;
- список подрядчиков;
- данные по новостройкам.

Непроверенные записи должны оставаться в статусе `CHECK`.

## Этап 3. Первый тест

Подключить 2–3 сотрудников и дать им задания из файла:

`docs/first-user-test.md`

Результатом теста должны быть не оценки, а реальные правки:

- что непонятно;
- что невозможно найти;
- какие слова сотрудники используют в поиске;
- каких ситуаций не хватает;
- какие поля лишние.

## Этап 4. Исправления

После теста:

1. Удалить лишние поля.
2. Добавить синонимы.
3. Упростить формулировки.
4. Заполнить самые частые контакты.
5. Назначить ответственных за разделы.
6. Перенести полезные заметки сотрудников в раздел `Практика`.
7. Отметить устаревшие или сомнительные записи.

## Этап 5. Запуск на офис

После исправлений таблицу можно открыть для всего офиса.

Рекомендуемый порядок:

1. Показать таблицу на планёрке.
2. Объяснить логику за 5 минут.
3. Дать 3 практических примера.
4. Попросить сотрудников пользоваться таблицей неделю.
5. Собрать обратную связь.

## Минимальные критерии успеха

- Новичок находит ответ по типовой ситуации за 1–3 минуты.
- Сотрудник понимает, куда обращаться по сложному вопросу.
- Менеджер получает меньше однотипных вопросов.
- В таблицу добавляются практические заметки.
- За первую неделю появляется не менее 10 предложений по улучшению.
- У каждой важной записи есть статус и ответственный.
- В публичном репозитории нет персональных данных клиентов.

## Что сделать после MVP

Если база используется сотрудниками, можно переходить к следующему этапу:

- собрать статистику самых частых поисков;
- выделить разделы для новичков;
- добавить мини-инструкции по сделкам;
- подготовить закрытый портал или CRM-модуль;
- связать базу с обучением и адаптацией сотрудников.
